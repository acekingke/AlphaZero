两幕高潮：

1. **loss 骗局**（第一幕）：loss 曲线完美，vs random 胜率 5%。揭示"训练指标 ≠ 能力指标"这一经典却仍被反复踩的坑。

## 骨架

### 第一幕 · 引子与第一轮破案（约 1200 字）

**开场**（约 250 字）：
- 第一句钩子："我训练了一个 AlphaZero。它打不过随机下棋。"
- 极简背景铺垫：6x6 Othello、自我对弈 → 训练 → Arena 接受/拒绝 的经典循环、训了三周、loss 曲线光滑
- 甩出症状：vs 随机 50 局，胜率 5%
- 自嘲收束："那一刻我开始怀疑的不是模型，是我对 ML 的整个认知。"

**第一轮破案**（约 900 字）：发现三个菜鸟坑
1. `num_epochs` 变量名骗了我：以为是"数据过 100 遍"，其实代码里只是"100 次梯度更新"。与 alpha-zero-general 对齐后才是真正的 10 个 epochs × 整个 buffer。
2. 训练 buffer 没有 sliding window：十几轮自我对弈的数据混在一起训练，陈年策略污染新策略。加上"只保留最近 20 轮"后才健康。
3. 没有外部 baseline：我每次只拿新模型对旧模型评估（Arena），从不和一个永远不变的对手打。这让整个训练系统变成一个自嗨回路。

**收束**：
- 三个修完 vs random 爬到 22%
- 最后一句留白："我以为故事结束了。"

**Takeaway 一**：**loss ≠ 能力；ML 里最便宜的"体检"是一个永远不变的外部对手。**

### 第二幕 · 13% 天花板（约 2000 字）

**新症状**（约 200 字）：继续训练十几轮，vs random 稳定在 13% 不动。

**三条线索全失败**（约 1000 字，每条 300–400 字）：
1. **调超参**：LR 从 1e-3 扫到 1e-4、Dirichlet α 从 0.3 调到 0.8、epochs 从 10 扫到 20 —— 结果几乎不变
2. **加网络容量**：channels 从 128 扩到 256、加了 dropout —— 训练 loss 更低了，但 vs random 胜率没动
3. **回代码找 bug**：修了几个 MCTS 的小问题（terminal value 的视角、backprop 跨 forced-pass 的玩家切换）—— 都是真 bug、修对了，但天花板依然不动

**关键转折：诊断实验**（约 500 字）：
- 写了两个小脚本：一个让网络绕过 MCTS 直接用 policy head 的 argmax 下棋，另一个走完整 MCTS（25 sims）
- 结果：**policy-only 43%，MCTS-25 sims 7%**
- "我盯着那个 43% 看了很久。如果让搜索介入反而让模型变弱，那答案只有一个 —— **搜索本身坏了**。"

**收束留白**（约 300 字）：顺势铺垫"算法对了，实现对了，那只剩一种可能 —— 我对搜索状态的组织方式是错的。"

### 第三幕 · 真相是一个数据结构（约 1800 字）

**逐行对比 reference 实现**（约 600 字）：
- 打开 alpha-zero-general 的 `MCTS.py`，最顶上 10 行就是全部状态：6 个 dict
  ```python
  Qsa = {}  # (s, a) -> Q
  Nsa = {}  # (s, a) -> 访问次数
  Ns  = {}  # s -> 总访问次数
  Ps  = {}  # s -> 先验策略
  Es  = {}  # s -> 终局标记
  Vs  = {}  # s -> 合法动作掩码
  ```
- 我的实现：Node 树，每个节点持有 children 列表和自己的 Q/N，从 root 向下递归搜索

**为什么这是结构性错误**（约 700 字）：
- Othello 里同一个局面（比如开局三步之后）可以由多种下法顺序到达
- tree 里：每种到达方式都创建一个独立的 Node。统计数据**不共享**
- dict 里：key 是棋盘 bytes（`board.tostring()`），同一个局面不管从哪条路径到达，统计都累积到同一个 entry
- 量化冲击：25 sims/move 下，tree 每个独立 Node 也许只被访问 1–2 次，估值方差极大；dict 下同样 25 次预算被分摊到更少的 unique state，单个 state 能积累 5–10 次访问。**信号/噪声比被拉高一个数量级**
- 还有一个容易忽略的细节：alpha-zero-general 在 Arena 40 局之间**不清空** dict，常见开局位置跨局累积上千次访问，有效搜索深度远大于名义 25 sims。我的 tree 实现每局重建，这个红利完全没吃到

**重写后的冲击**（约 400 字）：
- 把 MCTS 整个推倒，按 6 个 dict 的形态重写。`Node` 类保留成空 shim 以免旧的 import 炸掉
- 第一次跑 v20：第 **8 个迭代**的 Arena 之后，vs random 直接打到 **100%**
- 前面调了三周的所有超参、加的网络容量、修的代码 bug，都没这一行 `s = board.tostring()` 管用

**收束**：
- **Takeaway 二**：在对抗搜索里，"同构局面"必须被识别；**数据结构是算法的一部分，不是"实现细节"**
- **Takeaway 三**：当调参 + 代码 review 都无效，怀疑你的**数据表示**

### 尾声 · 三条我愿意带进下一个项目的教训（约 500 字）

1. **永远有一个不变的外部 baseline**。随机对手也行，只要它不变。loss 下降、Arena 胜率这些都是"照镜子"，外部 baseline 才是体重计。
2. **复现时别信变量名**。`num_epochs` 可以不是 epochs，`best_model` 可以不是 best。照着 reference 实现逐行对齐，是最没面子但最有效的动作。
3. **当调参和 review 都无效，怀疑你的数据表示**。算法教科书讲的是"它应该怎么算"，数据结构决定了"它实际在算什么"。两个 MCTS 实现可以算法完全等价、渐近复杂度相同，但一个能训出 100% 胜率、另一个永远卡在 13% —— 区别只是用 dict 还是 tree。

- 一个懂深度学习但没做过 AlphaZero 的读者，读完能清楚说出两件事：
  1. 为什么只看 loss 不够、怎样花最少的成本加一个外部 baseline

- 至少一张图具备"独立传播"能力（最可能是 vs random 爆点曲线图）
